# 0.5 deep audit — second pass over the container split

**Date:** 2026-07-12 (overnight autonomous run) ·
**Branch:** `0.5-containers`, `f260fe7` → `b1a0a1c` (7 commits) ·
**Test count:** 48 → 94 ·
**Findings:** 15 (9 fixed on the branch, 5 deferred to 0.6, 1 observation)

The split had already been built and smoke-tested; this pass hunted the
class of defect that survives a first verification: stale callers left by
refactors, semantics that drift during a move, and behavior only real
hardware exposes. Ground rules: one sanctioned live watch (skipjack), one
watch explicitly off-limits (lenok, weak cell), production v0.4 untouched.

## Method

Four independent probes, in order of increasing realism:

1. **Static integrity.** A `symtable`-based undefined-name sweep and an
   unused-import sweep over every module; a cache-bust parity table
   comparing all 26 routes against `main`; a hunt for references to names
   the split had moved.
2. **New test suites, validated against planted bugs.** Before trusting a
   new test, each check was run against a deliberately re-introduced copy
   of the historical bug it exists to catch (a dead onclick handler, a
   missing element id, a drifted API path). A test that cannot fail is
   decoration.
3. **Tri-mode live comparison.** Production v0.4, 0.5-monolithic and
   0.5-split served the same fleet simultaneously on different ports; their
   status documents were deep-diffed and every byte of difference explained
   (two process-lifetime artifacts, zero regressions). The full watch-action
   surface then ran against a live watch over the split's RPC path.
4. **Failure injection on real hardware.** Client disconnects mid-stream,
   backend killed mid-operation, 45 invalid tokens hammered at the gate,
   port conflicts, container stops, secret rotation — each expected
   recovery verified, not assumed.

## What the audit caught

### Containers (would have failed the reviewer's first trial)

- **The hardened frontend never started** (F12, fixed `14a7829`):
  `ReadOnly=true` plus podman's default secret mount is impossible — the
  `/run/secrets` mountpoint cannot be created on a read-only rootfs. The
  secret now mounts into `/tmp` (tmpfs). Found only by actually running
  the quadlet's exact flag set.
- **The reviewed host-adb design is unimplementable as specified** (F13,
  fixed `4ef63cc`): the host's adb server binds `127.0.0.1` and is
  unreachable from the container network — proven with both an adb probe
  and a raw TCP connect. The documented alternative (`adb -a`) would
  listen on the LAN, defeating the isolation the split exists for. The
  backend now runs its own adb server against the passed-through devices;
  an `ADB_SERVER_SOCKET` escape hatch remains for firewalled hosts.
- **Both containers ignored SIGTERM** (F14, fixed `b1a0a1c`): PID 1 gets
  no default signal dispositions, so every stop burned the runtime's
  10-second kill timeout. Explicit handlers; stops now take ~450 ms.

### Structure and hygiene

- The background cache warmer lived in the frontend module and the backend
  imported it from there, quietly breaking the architecture document's
  stated rationale (F3, fixed `2c9ddb3` — moved to `ops`, which simplifies
  the documented seams).
- `serve-backend` died with a raw traceback on an occupied port (F7,
  fixed `75ff5cd`).
- Dead imports in three modules (F1/F2/F5) — one of them created by the
  warmer move *during this audit* and caught by the new static suite
  within the hour it was written.

### Found on hardware, deferred to 0.6 (behavior decisions)

- **Ending a workbench leaves the watch running on battery** (F8): the
  cleanup cuts port power without the graceful shutdown that drain and
  charge perform. Discovered live — the sanctioned watch was found
  invisibly draining after "End checkout", which then made a poweroff
  probe report `adb_shutdown: false` (honestly: the watch wasn't on adb).
  Likely explains historical "watch is still on?!" reports. One-line fix,
  but it changes end-of-checkout semantics — maintainer's call.
- **Split mode needs a "backend unreachable" UI state** (F6): with the
  backend down, the status endpoint returns an error envelope the page
  cannot render.
- **The mass-poweroff mystery got a ground-truth protocol** (F11): a solo
  delivered halt was proven real — power off, re-power, then
  `/proc/uptime` reads 21.7 s (genuine cold boot; time-to-adb 37 s). The
  `uptime` *command* is untrustworthy on at least one watch (reported
  "up 109 days"; F9) — forensics must read `/proc/uptime`. The original
  multi-watch scene needs a repro with per-watch uptime forensics, and F8
  may account for much of it.
- One unexplained cold boot during a VBUS-transition window (F10) — watch
  for recurrence. Sustained token attacks produce a visible systemd
  restart loop (F15) — acceptable, documented.

## What was verified working (no action)

Status documents byte-identical between 0.5 modes; v0.4 deltas fully
explained. Control Center stable fields identical across all three modes
on a live watch. Screenshot as base64 over RPC decodes to a valid JPEG.
Screen force-on/release, clock sync, notification, buzz, WiFi toggle
round-trip with state restore. Onboarding stream survives a client
disconnect and completes server-side. Backend killed mid-workbench:
frontend degrades to a clean error, restarted backend auto-resumes the
operation, stop leaves zero residue. 24/24 concurrent status requests.
Token gate: 41 rejections logged with escalating backoff, listener
shutdown at threshold, nothing leaked on the wire. Secret rotation
end-to-end. Fleet returned to rest; the off-limits watch was never
powered.
